## The six drafter-flagged spots (extra scrutiny requested)

1. **ML-DSA-65 role** — VERIFIED against spec. The primer (§6 "Minting signatures: ML-DSA-65") attributes ML-DSA-65 exclusively to minting/block-reward signatures, with spends authorized via CLSAG. This matches whitepaper §4 ("Minting Signatures": "Block rewards (minting transactions) use ML-DSA-65 signatures since the minter's identity is public and must be verifiable long-term"), the §4 hybrid-rationale table, §2 ("ML-DSA-65 for minting: Block producer identity is public and must be unforgeable long-term"), and §5's transaction-type table (Minting → ML-DSA-65; Private → CLSAG). **Code divergence noted (major finding F2, operator-facing, not a primer defect)**: the live `MintingTx` in `botho/src/block.rs` carries *no* ML-DSA signature (minter identity binds via the RandomX PoW preimage over the minter's view/spend keys), while `botho/src/transaction_pq.rs` uses ML-DSA-65 to sign transaction *inputs* — the opposite allocation. The whitepaper is the declared oracle and the primer's own preamble says "when the two documents seem to disagree, the whitepaper wins," so this fires no flag against the primer; it is a whitepaper↔implementation gap the operator should reconcile.
2. **Bit-security levels** — VERIFIED. "NIST's middle security category" for ML-KEM-768 is true (FIPS security category 3, the middle of the five categories; whitepaper §3 itself labels ML-DSA-65 "security level 3"). Declining to quote §3's "approximately 192-bit / 128-bit security against quantum attacks" figures is a prudent hedge, not a contradiction — the primer asserts nothing §3 denies.
3. **Capstone fee "a few tens of nano-BTH"** — VERIFIED, arithmetic and units check out. Consensus floor per the spec's own worked derivation (§7, `minimum_fee_dynamic_with_outputs`): 1 pico-BTH/byte × ~5,000 bytes × factor 1 × output-penalty min(2,10)² = 4 → **20,000 pico-BTH = 20 nano-BTH** ("a few tens" ✓). Units are picocredit-canonical (1 BTH = 10¹² pico); no repeat of the historical 1000× fee-unit error. The ~5 KB size for 2-in-2-out matches the Parameter Justification appendix verbatim ("<5 KB for 2-in-2-out"). One lossy phrasing (minor finding F3): the primer's inline formula "base rate × size × factor 1" omits the ×4 output-count penalty that is precisely what lifts the result from ~5 to ~20 nano — the stated *magnitude* is right, the stated *formula* is incomplete.
4. **Demurrage gloss** — VERIFIED as lossy-but-true, confirmed against both spec and code. Primer: "a charge proportional to the value moved, the time held, and the cluster factor… Factor-1 coins are exempt." Code (`cluster-tax/src/demurrage.rs`): `charge = value × rate × (factor − 1)/(max_factor − 1) × elapsed/blocks_per_year` — proportional to value ✓, elapsed time ✓, and *affine* in factor with an exact zero at factor 1 (the primer states the factor-1 exemption explicitly, so no false belief is created). Whitepaper §7's compressed phrasing ("proportional to the spent value, the annual demurrage rate, and the coins' cluster factor") omits the elapsed-time term; the primer is *more* faithful to the implemented mechanism than the spec's own gloss, and contradicts neither.
5. **"Two-of-forty possible inputs"** — lossy-but-true (minor finding F4). Nothing in §5 forbids the two rings sharing decoy members (canonical ordering is enforced *within* a ring; the two real inputs are necessarily distinct via distinct key images), so 40 is an upper bound on the observer's candidate set, not a guarantee. In practice, independent sampling from a large output set makes overlap negligible, and the belief a reader carries away — "each spent coin hides among 20 candidates" — is exactly the spec's guarantee. Suggested tightening for revision: "two inputs, each hidden among twenty candidates."
6. **RandomX "roughly fair odds per watt"** — lossy-but-true (minor finding F5). The whitepaper's actual claims are ASIC resistance ("CPU-friendly PoW: Algorithm resists ASIC development (RandomX-based)", §7/§10), linear reward scaling ("No economy of scale"), and solo-mining viability. "Roughly fair odds per watt" for a laptop extends slightly past what the spec asserts (per-watt efficiency still varies across CPU classes), but the hedge ("roughly") and the carried-away belief — commodity hardware is not structurally disadvantaged and hashpower buys no consensus say — match the spec's intent. Not a contradiction; flag-free.
