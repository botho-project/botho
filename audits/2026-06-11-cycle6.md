# Internal Security Audit Report - Cycle 6

**Date**: 2026-06-11
**Auditor**: Internal (Claude Code assisted)
**Scope**: Consensus-economics changes since cycle 5 (lottery emission routing, spend-time demurrage, fixed-point fee/difficulty math, cluster-wealth fee enforcement, protocol-version disconnect) + block-acceptance path + dependencies
**Commit**: `620f359` (audit start; fixes applied during the audit are noted inline)

---

## Executive Summary

Cycle 6 audited the ~2,250 lines of consensus-critical code added since cycle 5
(cluster-tilted lottery with carryover pool, spend-time demurrage, progressive
fees on global cluster wealth, integer fixed-point conversions, protocol-version
peer disconnect). The new *mechanism* code is largely well built: integer-only
draw math, a documented anti-grinding payout cap, a single shared candidate
function for proposer and validator, and an overflow-analyzed demurrage charge.

However, following these mechanisms into the block-acceptance path
(`Ledger::add_block`) exposed that **most economic consensus rules are not
enforced when a block arrives from the network**. Blocks are checked for
height, prev-hash, self-declared PoW, lottery accounting, key images, and ring
signatures — and nothing else. This yields five Critical findings, including
accepting blocks at any self-declared difficulty, unvalidated minting rewards,
unverified ring membership (counterfeit inputs), and an unverified `tx_root`.
A recurring meta-pattern: **twice, consensus-hardening fixes landed in modules
the running node does not execute** (the 4a7899e float-elimination difficulty
controller, and the `transaction/core` progressive-fee validators).

These gaps predate or accompany the new economics work and were not visible to
cycles 1–5, which focused on cryptography, wallet, and network layers. None of
this is currently exploited (single-operator testnet), but **all Critical items
must land before any testnet where third parties mine or relay blocks** — and
they are consensus-breaking changes, so they should ride the already-planned
coordinated testnet reset.

## Fixes Applied During This Audit

1. **Lottery payout suppression** (was High) — `validate_block_lottery` now
   rejects no-winner blocks when the deterministic draw would have produced
   winners (`botho/src/consensus/lottery.rs`). Previously a producer could omit
   the (predictable) draw whenever it didn't favor them, carry the pool over,
   and harvest it later. Regression test added
   (`test_validate_block_lottery_rejects_suppressed_draw`).
2. **Protocol-version check failed open** (was Low) — an unparseable agent
   string now counts as consensus-incompatible (disconnect with `0.0.0`
   sentinel) instead of bypassing the check
   (`botho/src/network/discovery.rs`, `ProtocolVersion::consensus_incompatibility`
   + unit test).
3. **Dependency upgrades** — 9 of 15 `cargo audit` hits resolved:
   `bytes` 1.11.1, `quinn-proto` 0.11.14, `rkyv` 0.7.46, `time` 0.3.47,
   `rustls-webpki` 0.103.13, and **`ml-dsa` 0.1.0-rc.2 → 0.1.1**
   (RUSTSEC-2025-0144, timing side-channel in ML-DSA decomposition — this is
   the PQ transaction-authorization signature). The ml-dsa API changed;
   `crypto/pq/src/sig.rs` migrated with the 4032-byte secret-key format
   preserved. All crypto-pq tests pass.

## Critical Findings (all open)

### [CRITICAL] C1: Blocks accepted at self-declared difficulty

**Location**: `botho/src/block.rs:88-92` (`is_valid_pow`), `botho/src/ledger/store.rs:397-402`
**Status**: Open

`is_valid_pow()` compares the PoW hash against **the header's own `difficulty`
field**. `Ledger::add_block` never compares `header.difficulty` to the chain's
expected difficulty. `validate_minting_tx` (which does check it,
`consensus/validation.rs:154-161`) runs only in the SCP proposal path — blocks
arriving via gossip/sync (`add_block_from_network`) skip it entirely.

**Impact**: An attacker fabricates a chain with `difficulty` set to trivial
values and arbitrary content; syncing nodes accept it at zero PoW cost.

**Fix**: In `add_block`, require `header.difficulty == expected difficulty from
chain state` (and recompute expected difficulty deterministically — see C5).

### [CRITICAL] C2: Minting reward and timestamp unvalidated at block acceptance

**Location**: `botho/src/ledger/store.rs:459` (reward used as-is), `store.rs:380-442` (no timestamp checks)
**Status**: Open

`add_block` accepts `block.minting_tx.reward` without recomputing it from the
emission schedule (`calculate_block_reward`). The reward feeds `total_mined`,
the coinbase UTXO, and the lottery payout cap. Timestamps are not checked for
monotonicity or future bounds.

**Impact**: A block producer claims any reward → unlimited inflation accepted
by every validator.

**Fix**: Recompute the expected reward and emission share from chain state in
`add_block`; reject mismatches. Add timestamp sanity rules (median-time-past or
similar).

### [CRITICAL] C3: Ring members not verified against the UTXO set in block validation

**Location**: `botho/src/ledger/store.rs:890-906` (`verify_transaction`), `botho/src/transaction.rs:731-769`
**Status**: Open

Block-level transaction verification checks key images and CLSAG signatures
only. The ring members (target keys + **public amounts**) are taken from the
transaction itself and never resolved against `utxo_db`. The mempool does
resolve them (`validate_clsag_inputs`) — but blocks bypass the mempool.

**Impact**: A producer fabricates a ring member with a key they control and an
arbitrary amount; CLSAG verifies (they know the private key, and balance is
checked against the fabricated amount) → counterfeit value enters circulation.

**Fix**: In block validation, resolve every ring member against the UTXO set
and require the stored `(target_key, amount, creation height)` to match the
transaction's claimed ring data.

### [CRITICAL] C4: `tx_root` never verified; transaction list not bound to PoW

**Location**: `botho/src/block.rs:549` (`compute_tx_root`, private, construction-only), `botho/src/ledger/store.rs` (no check)
**Status**: Open

`header.tx_root` is computed at block construction and included in the header
hash, but `add_block` never recomputes it from `block.transactions`. Lottery
outputs/summary are not header-committed at all (they are bound only via
validator re-derivation of the draw).

**Impact**: Anyone relaying a block can swap the transaction list under a valid
header/PoW (substituted txs still need valid signatures and fresh key images,
but inclusion/exclusion control and fee capture transfer to the attacker; with
C1–C3 open the combined effect is total).

**Fix**: Verify `header.tx_root == compute_tx_root(&block.transactions)` in
`add_block`; consider committing lottery outputs and the minting tx hash too.
Note the current `compute_tx_root` is a flat hash of tx hashes, not a Merkle
tree — fine for binding, but document it.

### [CRITICAL] C5: Float-elimination fix landed in dead code; live difficulty path is still f64

**Location**: live: `botho/src/block.rs:982-994` (`EmissionController::adjust_difficulty`); dead: `cluster-tax/src/monetary.rs:484-610` (`DifficultyController`, integer)
**Status**: Open

Commit 4a7899e converted the cluster-tax `DifficultyController` to u128
basis-point math — but that controller is only reachable through
`MonetarySystem`, which has no callers in the node. The difficulty actually
persisted to chain state and hard-validated (`consensus/validation.rs:155`,
minting txs rejected on mismatch) is computed via `f64` ratio/clamp/cast.
The two controllers also use **opposite difficulty conventions** (block.rs:
lower = harder; cluster-tax: higher = harder).

**Impact**: Exactly the platform-divergence bug class 4a7899e set out to
eliminate, in hard-validated consensus state. (IEEE basic ops make divergence
unlikely in practice, but the no-fork policy makes "unlikely" insufficient.)

**Fix**: Port the integer basis-point computation into
`EmissionController::adjust_difficulty`; delete or wire up
`MonetarySystem`/`DifficultyController` so there is exactly one controller.

## High Findings

### [HIGH] H1: Demurrage and progressive fees are mempool policy, not consensus rules

**Location**: enforcement only at `botho/src/mempool.rs:652-735`; absent from `store.rs:484-516`, `consensus/validation.rs:226-261`, SCP validity callback
**Status**: Open

The progressive minimum fee and the spend-time demurrage charge (the
load-bearing component of the validated Gini mechanism) are computed once, at
mempool admission, on whichever node first sees the tx. Blocks containing
under-fee transactions validate everywhere — the only block-level floor is the
static `MIN_TX_FEE`. A miner includes their own txs fee-free; wealthy clusters
self-mine (or pay miners out-of-band) and never pay demurrage.

**Fix**: Recompute `base_minimum_fee + demurrage_charge(...)` per tx in block
validation from chain state at the block's height. **Prerequisite**: H3/M1 —
the current minimum-fee inputs include node-local congestion state and f64;
promoting them as-is would create chain splits.

### [HIGH] H2: Spender-chosen decoys defeat the demurrage clock and the cluster factor

**Location**: `cluster-tax/src/demurrage.rs:108-123` (`ring_elapsed_centroid`), `botho/src/mempool.rs:59-78, 684-701, 222-256`
**Status**: Open

Demurrage elapsed time is the value-weighted centroid of **spender-selected**
ring members; the cluster factor comes from **spender-authored** output tags
checked only by a cosine-similarity (≥0.7) plausibility test against the ring
centroid. Selecting fresh, high-value, background-tagged decoys both dilutes
the age centroid (up to ~ring-size reduction) and drags the tag centroid toward
background → factor 1000 → demurrage = 0 and fees at the 1x background rate.
`validate_cluster_tag_inheritance` only rejects tag *inflation*, not
evaporation, and is not called by the node.

**Fix options**: enforce a decoy-sampling age distribution; compute the charge
from a high quantile of ring age rather than the mean; floor the factor at the
ring-centroid-implied factor; make the plausibility check integer + consensus.

### [HIGH] H3: Difficulty/emission state persisted non-atomically with the block

**Location**: `botho/src/node/mod.rs:233-290`
**Status**: Open

`add_block` commits in one LMDB txn; `update_emission_state` (difficulty, epoch
counters) commits in a second. A crash between them leaves height advanced with
stale difficulty; on restart the node computes different difficulty than peers
forever (difficulty is hard-validated). One unlucky crash = permanent
divergence.

**Fix**: fold the emission-state update into the `add_block` write transaction
(it is a pure function of the applied block).

### [HIGH] H4: Lottery candidate cap excludes most UTXOs at scale and is grindable

**Location**: `botho/src/ledger/store.rs:1851-1878` (`MAX_LOTTERY_CANDIDATES = 10_000`, first-N in LMDB key order)
**Status**: Open

Once >10k eligible UTXOs exist, only the lexicographically lowest UTXO IDs ever
participate in the lottery — permanently. UTXO IDs are `tx_hash || index`, and
a low tx hash can be vanity-ground cheaply, buying a permanent seat in the
candidate window while everyone else is excluded. Redistribution silently stops
reaching most holders.

**Fix**: deterministic per-block sampling instead of a fixed prefix — e.g.,
start the iteration at the key offset derived from the lottery seed
(`H(prev_hash, height)`), wrapping around, still capped at 10k. Identical for
proposer/validator, unbiased over time, and grinding a hash position no longer
guarantees membership.

### [HIGH→FIXED] Lottery payout suppression — see Fixes Applied (#1)
### [HIGH→FIXED] ml-dsa timing side-channel (RUSTSEC-2025-0144) — see Fixes Applied (#3)

## Medium Findings

### [MEDIUM] M1: f64 and node-local state throughout the enforced minimum-fee path
`cluster-tax/src/fee_curve.rs:196-207` (`f64::powf` output penalty — libm-dependent,
genuinely platform-divergent), `dynamic_fee.rs:65-135` (f64 EMA of locally
observed congestion, reset on restart), `mempool.rs:101,149-256` (f64 cosine
similarity), demurrage validated against the node's own tip height. Today this
only causes admission/relay divergence; it becomes a chain-split bug the moment
H1 is fixed naively. Fix before/with H1: integer output penalty, consensus
minimum defined from deterministic inputs only, congestion EMA demoted to relay
policy.

### [MEDIUM] M2: Cluster wealth is monotonically non-decreasing (cumulative volume, not holdings)
`store.rs:1303-1336`: every output ever created adds to `cluster_wealth_db`;
nothing decrements (impossible under ring privacy). "Wealth" = lifetime tagged
volume; active commerce clusters ratchet toward the 6x factor regardless of
holdings, while sophisticated spenders zero it out via H2. Deterministic, so
economic-only. Needs a design answer (decay schedule, or accept and document).

### [MEDIUM] M3: Rebuild vs incremental wealth arithmetic mismatch
`rebuild_cluster_wealth_index` uses unchecked `+=` (`store.rs:1480`); the
incremental path uses `saturating_add` (`store.rs:1327`). At the saturation
boundary (reachable eventually, since M2 makes wealth cumulative-forever) a
rebuilt node diverges from an accumulated one. Use `saturating_add` in both.

### [MEDIUM] M4: All fees booked as burned while 80% is redistributed
`store.rs:461-463`: `total_fees_burned += block_fees`, but only the 20% burn
share is destroyed; lottery payouts mint UTXOs from value booked as burned.
5x overstatement in consensus-adjacent state (`META_FEES_BURNED`,
`epoch_burns`). No consensus effect today (live difficulty ignores burns), but
any future supply logic inherits the error.

### [MEDIUM] M5: Difficulty responds to tx-count, not time — producer-skewable
`block.rs:922-997`: epochs are 1000 transactions; the adjustment reduces to
`20 / avg_txs_per_block`. Producers skew difficulty for free by stuffing or
starving blocks. Also `MAX_DIFFICULTY = INITIAL_DIFFICULTY` means PoW can never
get easier than genesis. Revisit alongside C5.

### [MEDIUM] M6: Lottery duplicate-roll discard skews payouts
`cluster-tax/src/lottery.rs:275-303`: rolls are taken modulo the total weight
*including* already-selected candidates; a roll landing on a used index selects
nobody, and the shortfall funnels to the **last** winner via the remainder
fix-up. Deterministic (consensus-safe) but increasingly distorting as weight
concentrates. Fix: re-roll over remaining weight (recompute the modulus
excluding used indices).

### [MEDIUM] M7: Consensus-critical DB checks swallow errors (fail-open)
`store.rs:893` (`if let Ok(Some(..))` on `is_key_image_spent` — a DB error
passes the double-spend check), `store.rs:940` (same pattern in
`record_key_image`'s existence check), `mempool.rs:68` (`get_cluster_wealth`
error → wealth 0 → lower fee). Propagate errors; never default-allow on a
consensus check.

## Low Findings

### [LOW] L1: No ban/backoff after version-incompatible disconnect
`discovery.rs`/`run.rs`: a disconnected peer can redial immediately; during a
coordinated upgrade the entire old network reconnect-loops against new nodes.
PoW challenge + per-IP caps bound the cost. Add a short (e.g., 1h) ban window.

### [LOW] L2: Remaining dependency advisories (6)
- `hickory-proto` 0.25.2 ×2 (DNS DoS; one advisory unpatched upstream, the
  other needs `hickory` 0.26 via a libp2p bump)
- `rustls-webpki` 0.102.8 ×4 (CRL/name-constraint flaws; pinned by
  `rustls 0.22` ← `sentry 0.34` — needs a sentry upgrade)
All are DoS/cert-validation in telemetry/DNS paths, not consensus.

### [LOW] L3: Dead `DifficultyController` hazards (if ever wired up)
`monetary.rs:546-549` divide-by-zero on pathological config; `monetary.rs:591`
truncating u128→u64 cast; unchecked `+=` on counters. Resolve with C5 (delete
or fix).

## Informational

- **I1**: Protocol versions are self-reported via libp2p identify — the
  disconnect is upgrade hygiene, not a security boundary; lying peers are
  excluded only by block validation. Document as such.
- **I2**: Spent UTXOs remain lottery-eligible forever (ring privacy makes spent
  status unknowable). Inherent; interacts with the "everyone parks" watch item.
- **I3**: `transaction/core` progressive-fee validators
  (`compute_progressive_fee`, `validate_cluster_tag_inheritance`) have no
  callers — second instance of the dead-code pattern (see C5). Delete or wire up.
- **I4**: Multi-input CLSAG balance verification looks structurally wrong:
  `transaction.rs:1043-1053` requires each input's real amount to equal the
  full `outputs + fee` total, so honest multi-input txs cannot validate.
  Functional defect (prevents rather than permits theft); single-input balance
  is enforced exactly, which is currently the *only* block-level balance check.
- **I5**: Demurrage edge cases verified safe: charge is a fee floor (no
  underflow), u128-widened products (max ~2^109), saturating elapsed math,
  factor-1 exemption, consistent floor rounding, height-based (not timestamp)
  accrual.
- **I6**: No fork-choice/reorg handling is visible in `add_block` (exact
  next-height only). Whether SCP finalization makes competing blocks at the
  same height impossible needs explicit verification and documentation —
  if gossip races SCP, first-received-wins would partition nodes.

## What Is Implemented Correctly (spot-verified)

- `demurrage_charge`: pure integer, overflow-analyzed, churn-invariant, tested.
- Demurrage elapsed time derives from block heights, not timestamps.
- Integer LUT cluster-factor curve and integer sigmoid fee rate: exact,
  monotonic, bounded.
- Lottery: integer permille pool accounting, payout cap with explicit
  anti-grinding argument (regrind costs a full PoW for ≤1 reward of payout),
  validator re-derivation of the draw, single shared candidate function with
  documented deterministic ordering, f64 confined to research-only modes.
- 306ad83 closes naive split-evasion: fees weigh output tags against ledger-
  global cluster wealth.
- Mempool balance/output sums use `checked_add`; in-block double-spends are
  caught by `record_key_image` inside the write txn.
- Block + lottery-pool state commit in a single LMDB transaction.

## Priority Plan for Testnet Release

1. **Block-acceptance hardening (C1–C4)** — one coherent change to
   `Ledger::add_block`: expected difficulty, expected reward + emission share,
   tx_root recomputation, ring-member resolution, timestamp rules. Blocks the
   public testnet; consensus-breaking, so it rides the planned reset.
2. **Single integer difficulty controller (C5, with M5, L3, I3 cleanup)**.
3. **Consensus fee floor (H1) after deterministic-input rework (M1)**; decoy
   policy for demurrage (H2) — design decision required.
4. **Atomic emission-state persistence (H3)** — small, do immediately.
5. **Lottery candidate sampling (H4) and duplicate-roll re-roll (M6)** —
   consensus-breaking, ride the same reset.
6. Process rule (meta-pattern): every consensus-math change must demonstrate
   reachability from `add_block`/SCP validity — twice now, hardening landed in
   dead code.
